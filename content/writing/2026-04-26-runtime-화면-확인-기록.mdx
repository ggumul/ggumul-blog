이번에는 문서나 코드만 보고 끝내지 않고, 실제로 켜지는 화면을 기준으로 다시 봤습니다. 대상은 브라우저에서 바로 플레이할 수 있는 Hanoi와, iOS Simulator에서 실행한 Wanderer 모바일입니다.

처음에는 “앱이 켜진다”만 확인하면 충분해 보였습니다. 그런데 실제 화면을 이어서 보니 버튼을 누른 뒤 다음 상황이 바로 이해되지 않는 구간이 있었고, 한 판의 리듬도 생각보다 자주 끊겼습니다. 이 기록은 그 지점들을 다음 작업의 기준으로 남긴 글입니다.

서버 쪽인 `wanderer-was`는 화면을 가진 프로젝트가 아니라서 별도 영상으로 묶지는 않았습니다. 대신 테스트를 돌려서 현재 코드가 깨지지 않는지만 확인했습니다.

## 확인한 것

- Hanoi: 로컬 브라우저에서 실행 후 퍼즐 조작 확인
- Wanderer 모바일: Flutter 환경 파일 생성 후 iOS Simulator에서 앱 실행 확인
- Wanderer 서버: Gradle 테스트 통과 확인

이번 기록에서 중요한 기준은 “실제로 사용자 화면까지 도달했는가”였습니다. 그래서 화면이 없는 서버는 실행 영상이 아니라 테스트 결과로만 분리했고, 화면이 있는 프로젝트만 영상으로 남겼습니다.

## Hanoi

Hanoi는 웹앱으로 실행했습니다. 로컬 서버를 띄운 뒤 브라우저에서 접속했고, 원반을 선택하고 다른 기둥으로 옮기는 기본 조작까지 확인했습니다. 브라우저 콘솔 오류는 나오지 않았습니다.

현재 화면 기준으로는 퍼즐 상태, 이동 횟수, 조작 버튼이 한 화면에서 확인됩니다. 아직 더 봐야 하는 부분은 난이도 선택 이후의 플레이 흐름과 모바일 화면에서의 손 조작감입니다. 이번 확인은 “앱이 켜지고 기본 조작이 된다”는 기준선까지입니다.

## Wanderer 모바일

Wanderer 모바일은 iOS Simulator에서 실행했습니다. 먼저 로컬 환경 설정 파일을 생성한 뒤 로컬 서버를 `perf` 프로필로 띄우고, Flutter 통합 테스트가 실제 앱 화면을 움직이도록 두었습니다. 이번에는 짧은 상태 컷이 아니라 로그인 이후 홈, 빠른 대전, 매칭 대기, 게임 화면, 종료 결과까지 이어지는 흐름을 다시 녹화했습니다.

영상 초반에는 홈/로비에서 캐주얼 빠른 대전을 시작하는 화면이 나오고, 이어서 매칭 대기, 게임 중 제출 단계, 이모트 패널, 종료 후 `다시 플레이`/`로비로` 선택 화면까지 이어집니다. 그래서 로그인 대기 화면이나 단일 프레임 조합이 아니라, 현재 앱이 실제로 한 판의 흐름을 어디까지 보여주는지 확인할 수 있습니다. 다만 이 영상은 로컬 확인 환경에서 촬영한 기록이므로, 운영 서버에서의 장시간 동시 접속 안정성은 별도 확인 대상으로 남깁니다.

## 서버와 화면 확인을 분리한 이유

Wanderer 서버는 테스트가 통과했지만, 서버 테스트 통과와 모바일 플레이 가능은 같은 의미가 아닙니다. 서버가 살아 있고 테스트가 통과해도, 모바일이 기대하는 이벤트 이름과 payload가 다르면 실제 플레이 흐름은 끊길 수 있습니다.

그래서 이번 기록에서는 서버를 “화면 시연 대상”으로 쓰지 않았습니다. 서버는 테스트 결과로만 확인하고, 사용자가 직접 볼 수 있는 화면은 모바일과 웹앱 영상으로만 남겼습니다.

## 지금 기준의 결론

Hanoi는 브라우저에서 기본 조작까지 확인된 상태입니다. Wanderer 모바일은 iOS Simulator와 로컬 서버를 붙여 홈에서 빠른 대전을 시작하고, 매칭 대기와 게임 진행, 종료 결과 화면까지 한 흐름으로 다시 확인했습니다. Wanderer 서버는 테스트 기준으로도 깨지지 않았고, 이번 모바일 영상의 로컬 실행 흐름에도 같이 사용했습니다.

다음에 이어서 볼 부분은 두 가지입니다. 하나는 Wanderer를 운영 서버/장시간 동시 접속 기준에서도 같은 흐름으로 확인하는 것, 다른 하나는 Hanoi를 모바일 브라우저 기준에서도 조작해 보는 것입니다. 이번 기록은 “현재 로컬 실행 화면으로 확인 가능한 실제 흐름”을 남긴 것입니다.
