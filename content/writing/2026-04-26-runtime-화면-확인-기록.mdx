이번에는 문서나 코드만 보고 정리하지 않고, 실제로 켜지는 화면을 기준으로 다시 확인했습니다. 대상은 브라우저에서 바로 확인할 수 있는 Hanoi와, iOS Simulator에서 실행한 Wanderer 모바일입니다.

서버 쪽인 `wanderer-was`는 화면을 가진 프로젝트가 아니라서 별도 영상으로 묶지는 않았습니다. 대신 테스트를 돌려서 현재 코드가 깨지지 않는지만 확인했습니다.

## 확인한 것

- Hanoi: 로컬 브라우저에서 실행 후 퍼즐 조작 확인
- Wanderer 모바일: Flutter 환경 파일 생성 후 iOS Simulator에서 앱 실행 확인
- Wanderer 서버: Gradle 테스트 통과 확인

이번 기록에서 중요한 기준은 “실제로 사용자 화면까지 도달했는가”였습니다. 그래서 화면이 없는 서버는 실행 영상이 아니라 테스트 결과로만 분리했고, 화면이 있는 프로젝트만 영상으로 남겼습니다.

## Hanoi

Hanoi는 웹앱으로 실행했습니다. 로컬 서버를 띄운 뒤 브라우저에서 접속했고, 원반을 선택하고 다른 기둥으로 옮기는 기본 조작까지 확인했습니다. 브라우저 콘솔 오류는 나오지 않았습니다.

<video controls preload="metadata" style="width: 100%; border-radius: 24px; border: 1px solid rgba(255,255,255,0.12); background: #050505;" src="/media/runtime-checks/hanoi-demo.mp4"></video>

현재 화면 기준으로는 퍼즐 상태, 이동 횟수, 조작 버튼이 한 화면에서 확인됩니다. 아직 더 봐야 하는 부분은 난이도 선택 이후의 플레이 흐름과 모바일 화면에서의 손 조작감입니다. 이번 확인은 “앱이 켜지고 기본 조작이 된다”는 기준선까지입니다.

## Wanderer 모바일

Wanderer 모바일은 iOS Simulator에서 실행했습니다. 먼저 로컬 환경 설정 파일을 생성한 뒤 Flutter 분석과 테스트를 통과시켰고, 이후 시뮬레이터에서 앱 화면을 캡처했습니다.

이번 확인에서 Wanderer는 최소한 모바일 앱으로 빌드되어 시뮬레이터 화면까지 올라오는 것을 확인했습니다. 다만 이 단계만으로 실제 게임 진행이 완성됐다고 말할 수는 없습니다. 서버 sync와 게임 흐름은 별도 계약 문제로 남아 있고, 그 부분은 이전에 정리한 sync 기록과 이어서 봐야 합니다.

## 서버와 화면 확인을 분리한 이유

Wanderer 서버는 테스트가 통과했지만, 서버 테스트 통과와 모바일 플레이 가능은 같은 의미가 아닙니다. 서버가 살아 있고 테스트가 통과해도, 모바일이 기대하는 이벤트 이름과 payload가 다르면 실제 플레이 흐름은 끊길 수 있습니다.

그래서 이번 기록에서는 서버를 “화면 시연 대상”으로 쓰지 않았습니다. 서버는 테스트 결과로만 확인하고, 사용자가 직접 볼 수 있는 화면은 모바일과 웹앱 영상으로만 남겼습니다.

## 지금 기준의 결론

Hanoi는 브라우저에서 기본 조작까지 확인된 상태입니다. Wanderer 모바일은 iOS Simulator에서 앱 실행 화면까지 확인된 상태입니다. Wanderer 서버는 테스트 기준으로는 깨지지 않았지만, 모바일과 서버를 붙여서 실제 게임 진행까지 검증한 상태는 아닙니다.

다음에 이어서 볼 부분은 두 가지입니다. 하나는 Wanderer 모바일과 서버의 sync 계약을 실제 실행 흐름에서 맞추는 것, 다른 하나는 Hanoi를 모바일 브라우저 기준에서도 조작해 보는 것입니다. 이번 기록은 그 전 단계, 즉 “현재 화면으로 확인 가능한 기준선”을 남긴 것입니다.
