여기서는 Wanderer 모바일과 서버의 sync 연결이 왜 바로 붙지 않았는지 정리합니다. 처음에는 서버 쪽 websocket 기능이 아직 준비되지 않은 것처럼 보였지만, 다시 확인해 보니 핵심 원인은 transport 부재가 아니라 상태 복구 계약의 차이였습니다.

## 문제

모바일 클라이언트는 reconnect 이후 `/app/room/sync/request`를 보내고, 서버가 `SYNC_SNAPSHOT` 이벤트 하나로 현재 방 상태를 다시 내려주는 흐름을 기대하고 있었습니다. 이 방식에서는 클라이언트가 “지금 기준 상태”를 한 번에 받아서 view model에 적용할 수 있어야 합니다.

반면 서버는 같은 상황을 하나의 snapshot으로 내려주기보다, room, game, turn submission 변경을 각각 이벤트로 발행하는 구조에 가까웠습니다. 서버 입장에서는 `ROOM.UPDATED`, `GAME.UPDATED`, `TURN_SUBMISSIONS_UPDATED`를 순서대로 보내면 충분해 보일 수 있지만, 모바일은 이 이벤트들을 조합해서 기준 상태를 복원하도록 설계되어 있지 않았습니다.

## 실제로 맞지 않았던 것

최신 `wanderer-was` 기준으로 실시간 연결 자체는 이미 구성되어 있었습니다. 연결 주소와 메시지 처리 코드도 있었고, 따라서 문제를 “서버에 실시간 연결이 없다”로 보는 것은 틀렸습니다. 연결이 가능해 보여도, 실제로는 이름, 데이터 모양, 상태 복구 방식이 함께 맞아야 합니다.

모바일 쪽 기대값을 단순화하면 다음에 가깝습니다.

```json
{
  "event": "SYNC_SNAPSHOT",
  "room": { "id": "...", "players": [] },
  "game": { "phase": "...", "turn": 1 },
  "submissions": []
}
```

서버 쪽 현재 흐름은 이 구조와 다릅니다. 서버는 이미 방, 게임, 제출 상태를 만들 수 있는 재료를 갖고 있지만, 모바일이 바로 이해할 수 있는 하나의 응답으로 묶어 보내는 호환 계층은 별도로 필요합니다.

## 선택지

첫 번째 선택지는 모바일을 서버의 세분화된 알림 방식에 맞게 바꾸는 것입니다. 이 경우 데이터를 읽는 코드, 저장소, 재접속 처리, 화면 상태 적용 흐름을 함께 손봐야 합니다. 장기적으로는 더 세밀한 처리가 가능하지만, 지금 당장 연결 기준선을 복구하기에는 수정 범위가 큽니다.

두 번째 선택지는 서버가 모바일이 바로 이해할 수 있는 복구 응답을 추가로 제공하는 것입니다. 서버의 내부 구조는 유지하되, 재접속 요청에 대해서만 방, 게임, 제출 상태를 한 번에 묶어 보내는 방식입니다. 현재 기준에서는 이 선택이 더 작고 안전합니다. 새로운 게임 상태를 발명하는 것이 아니라, 이미 있는 상태를 모바일 약속에 맞게 포장하는 작업에 가깝기 때문입니다.

## 판단

이번 문제는 enum 몇 개를 맞추는 수준의 불일치가 아닙니다. 모바일과 서버가 “sync”라는 같은 단어를 쓰고 있었지만, 한쪽은 snapshot 복구를, 다른 한쪽은 이벤트 스트림 갱신을 기준으로 삼고 있었습니다. 그래서 연결이 안 된 이유는 네트워크가 아니라 계약 해석의 차이였습니다.

우선순위는 서버 호환층을 추가해 모바일의 현재 구조를 살리는 쪽입니다. 그 다음에 실제 게임 진행 중에는 여러 상태 변화를 어떻게 이어서 적용할지 별도 작업으로 분리하는 편이 낫습니다. 재접속 복구와 진행 중 업데이트를 한 번에 재설계하면 수정 범위가 커지고, 실패했을 때 원인을 다시 나누기 어려워집니다.

## 이어서 볼 부분

이어서 볼 일은 세 단계로 나눕니다. 첫째, 서버 테스트에서 재접속 요청 뒤 복구 응답이 한 번 내려오는지 확인합니다. 둘째, 그 안에 방, 게임, 제출 상태가 모바일이 기대하는 이름과 형태로 들어가는지 고정합니다. 셋째, 모바일에서 다시 접속한 뒤 해당 상태를 받아 화면이 복구되는지 확인합니다.

이 글의 결론은 단순합니다. 실시간 기능은 연결 주소가 열려 있다고 끝나는 일이 아닙니다. 메시지를 받는 위치, 이름, 데이터 모양, 복구 시점, 실패 후 재진입 방식까지 같은 약속으로 맞아야 실제 플레이가 이어집니다. Wanderer의 우선 과제는 이 기준을 글과 테스트로 고정한 뒤, 모바일과 서버를 다시 같은 흐름에 올리는 것입니다.
